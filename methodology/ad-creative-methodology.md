# 广告创意方法论精简库

本文件用于 ad-creative-skill 的 Phase1 分析阶段。它把参考视频拆解、产品映射和故事方向池放进同一套判断语言中，避免只复述画面或套用表层题材。

这是默认执行版。原始全文方法论保存在 `methodology/full/`，用于补充解释、方法选择依据、案例机制和未来 Phase2 设计；它不覆盖本文件的 Phase1 执行边界。

## 默认调用顺序

1. 读取产品上下文：先看 `product-brief.md`，确认产品品类、真实玩法、核心玩法体验链路、首个可验证体验、可用资产、制作限制、合规限制和成功指标。
2. 判断任务类型：区分单条参考、同方向 mix、素材复盘、创意方向发散、分镜/prompt。当前 skill 的 Phase1 只处理参考拆解和故事方向池。
3. 做基础诊断：判断参考素材的钩子、冲突、承接桥、产品证明、素材定位、触发机制和可见反馈是否成立。
4. 选择方法卡片：按素材缺口选择 1 个主方法和最多 1 个辅助方法；不要为了套模板而选择方法。
5. 参考案例机制：学习 BGM、包装、强刺激、节奏画面、验证脚本结构或高风险吸睛机制，但只迁移结构，不照搬题材。
6. 保持 Phase1 边界：只输出候选故事方向；用户明确选定方向后，才进入分镜级脚本、prompt、出图和版本迭代。

如果任务是广告表达形式、素材结构、测试优先级或创意方向池，不能只根据当前产品录屏判断。必须同时结合当前产品玩法事实、当前产品自己的具体素材/素材记忆，以及根级 `competitors/` 模块中的同玩法竞品素材；竞品素材不能放进任何 `products/<product-id>/` 目录。

## 核心原则

| 原则 | 判断问题 | 失败信号 |
| --- | --- | --- |
| 钩子抓停留 | 前几秒是否有清楚的异常、压力、选择、爽点或疑问？ | 开头像功能介绍，用户没有继续看的理由。 |
| 承接桥 | 钩子的注意力如何转译到产品资产、玩法动作或产品幻想？ | hook 很强，但后续和产品体验断裂。 |
| 产品证明 | 用户能否相信下载后有对应体验？ | 只有噱头、特效或剧情，没有真实玩法证据。 |
| 素材即定位 | 素材在吸引哪类用户、过滤哪类用户？ | 只追 CTR，吸引大量低质量泛用户。 |
| 触发机制 | 关键变化为什么发生，能否 1-2 个画面看懂因果？ | 角色或资产凭空变强，爽点缺少来源。 |
| 可见反馈 | 用户动作是否立刻产生视觉、数值、状态或情绪反馈？ | 功能被讲清楚了，但没有动作欲望。 |

## 方法卡片选择矩阵

| 方法 | 适合问题 | 核心结构 | 检查问题 | 失效风险 |
| --- | --- | --- | --- | --- |
| 同源多结果法 | 开箱、抽卡、合成、进化、打造等有同一起点和不同结果的系统。 | 同一起点 -> 不同条件 -> 不同结果 -> 横向对比。 | 起点是否一眼看懂？每次变化后是否立刻给结果？结果差异是否足够明显？ | 只是展示差异，没有情绪推进；一次塞太多结果。 |
| 被低估资产逆袭法 | 弱变强、低星逆袭、废卡养成、普通宠物进化、失败角色翻盘。 | 被低估 -> 坏局 -> 隐藏成长线 -> 逐步变强 -> 完成逆袭。 | 弱点是否真实可见？成长是否有过程？逆袭是否由产品机制承接？ | 弱小资产太低质；逆袭过快或像剧情硬转。 |
| 短反馈链法 | 操作、升级、战斗、解谜、经营等需要证明爽感的素材。 | 动作 -> 立即反馈 -> 奖励/变化 -> 下一次动作。 | 每个关键动作是否有即时反馈？反馈是否可见且足够爽？ | 只讲系统，不展示动作和反馈。 |
| 对照组闭环法 | 逆袭、竞技、成长、复仇、排名提升、阵容对抗。 | 初始差距 -> 对手压力 -> 方法改变 -> 差距反转 -> 闭环证明。 | 对手是否服务主角成长？结尾是否完成对照闭环？ | 对手抢走产品卖点；对照关系不清。 |
| 夸张视觉刺激法 | 巨大化、进化、秒杀、爆炸、战力暴涨、Boss 压迫、大规模战斗。 | 强视觉异常 -> 解释触发 -> 结果兑现 -> 产品证明。 | 夸张是否服务产品卖点？视觉刺激后是否有承接桥？ | 只有特效，没有真实承接。 |
| 反常识结果互动法 | 选择题、孵化、合成、开箱、挑战、解谜、隐藏配方。 | 普通预期 -> 反常识选择/条件 -> 异常结果 -> 解释原因。 | 反常识是否能被快速理解？结果是否和产品机制有关？ | 反常识太随机，像噱头而非玩法。 |
| 挑战失败诱导法 | 闯关、解谜、跑酷、塔防、操作挑战、差一点通关。 | 明确目标 -> 接近成功 -> 关键失败 -> 用户产生纠错欲。 | 失败是否让用户觉得“我能过”？失败原因是否清楚？ | 失败太蠢或太惨，降低产品质感。 |
| 故事冲突驱动法 | 功能正确但观感平、玩法承接变成功能演示、战斗和养成缺情绪。 | 目标 -> 阻碍/压力 -> 行动 -> 假失败/升级危机 -> 核心卖点解决。 | 主角想要什么？失败会损失什么？结尾是否用产品卖点解决冲突？ | 冲突太弱、太复杂或和产品脱节。 |
| 素材即定位法 | CTR 还可以但 ROI、留存或付费弱；需要区分泛用户和核心用户。 | 目标用户 -> 长期动机 -> 素材信号 -> 平台学习 -> 后端验证。 | 这条素材吸引谁？哪些画面在筛用户？是否有低质量泛流量风险？ | 只追高 CTR；素材信号互相冲突。 |
| 触发机制可信化法 | 弱小单位变强、隐藏职业觉醒、合成升级、进化突破。 | 关键变化 -> 外部刺激/隐藏规则/资源消耗 -> 变化发生 -> 核心收益。 | 变化前后差异是否强？能量或资源从哪里来？为什么和产品功能有关？ | 解释太复杂；变化像凭空发生。 |

## 优秀案例拆解框架

| 机制 | 适合学习什么 | 使用边界 |
| --- | --- | --- |
| BGM 驱动 | 音乐决定情绪骨架、剪辑卡点和高潮位置。 | 分镜中要标注卡点；音乐不够记忆点时不要强行归类。 |
| 包装形式 | 用用户熟悉的内容壳降低理解门槛，如发布会、采访、真假测试、真人场景嫁接。 | 包装不能吞掉产品卖点，也不能替代产品承接。 |
| 强刺激钩子 | 通过视觉、声音、冲突、选择、危机或关系张力先抓停留。 | 必须通过承接桥转回产品幻想。 |
| 节奏与画面结合 | 让音乐、镜头、转场、动作和情绪递进一起推动观看。 | 不能只有炫技剪辑，缺少目标、冲突和兑现反馈。 |
| 验证过的脚本结构 | 提炼可复用创意骨架，例如目标-阻碍-反转-兑现。 | 学结构，不学表层题材和台词。 |
| 高风险吸睛机制 | 学习吸睛机制和边界定义，用于风险提示或负面案例训练。 | 不用低俗、误导或平台风险元素替代产品核心。 |

## Phase1 参考视频特殊规则

标准顺序：

```text
原视频场景变化拆解
-> 原视频底层结构和可迁移点
-> 候选故事方向筛选池
-> 人类认可某个方向
-> Phase2 分镜级脚本、prompt、出图和版本迭代
```

Phase1 只允许产出候选故事方向。可以拆解原参考视频的结构和关键帧，但不要提前为用户产品创建逐镜头 production storyboard（制作分镜）、出图 prompt、关键帧画面描述、过细时间码脚本或 `script-*` 文件夹。

## 正式输出最小要求

每次补写 `outputs/` 时，先简短说明：

- 本次调用了哪些产品上下文。
- 本次主要诊断问题是什么。
- 本次采用了哪些方法。
- 哪些方法被排除，为什么。

每个候选方向至少回答：

- 核心假设、hook、故事前提和冲突升级。
- 关键变化由什么触发。
- 承接桥如何把 hook 转译到产品幻想。
- 产品证明在哪里出现。
- 目标用户信号是什么，适合验证什么指标。
- 主要风险和需要人工判断的问题。
